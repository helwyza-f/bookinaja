# Bookinaja Mobile Stitch Master Prompt

Use this as the main master prompt in Stitch before generating any screens.

---

You are designing **Bookinaja Mobile**, a consumer + admin mobile product for a booking and operational platform used by gaming centers, entertainment venues, rental spaces, and service-based businesses.

Your job is not to make a generic operational app.

Your job is to create a mobile product that feels:

- emotional
- engaging
- personal and alive
- premium
- smooth
- delightful
- playful in the right places
- operationally sharp
- memorable enough that people want to talk about it

## Core product truth

Bookinaja is not just a utility.

It is the feeling of:

- finding a place to play or book with confidence
- seeing a live session feel active and real
- having the admin feel in control, fast, and modern
- turning operational work into something that feels premium, not boring

Useful is not enough.
The product must leave people feeling:

- guided
- calm
- in control
- rewarded
- connected to live activity

## Design direction

Do not design this like:

- a dashboard template
- a B2B SaaS admin panel squeezed into mobile
- a plain white app with random cards
- an MVP demo
- a clone of Duolingo, Linear, Uber, or Airbnb

Use them only as reference for:

- recognizability
- hierarchy
- motion quality
- emotional feedback
- polish

But the resulting product must feel uniquely **Bookinaja**.

## Brand personality

Bookinaja should feel like:

- a live signal
- a pulse of activity
- a command center
- a premium digital venue companion

Keywords:

- live
- signal
- momentum
- session
- pulse
- clean confidence
- modern hospitality
- premium operations

## Emotional design goals

Every important screen should answer:

1. What is happening right now?
2. What matters most right now?
3. What should I do next?
4. Does the interface feel alive when I touch it?
5. Does the interface feel worth remembering?

## Visual language

Create a distinct visual language for Bookinaja Mobile:

- more expressive than a plain business app
- more premium than a startup MVP
- more alive than a sterile admin console
- still clean, simple, and operational

### Visual principles

- Less verbose, more visual.
- More icon, less copy.
- More layout, less stacked rectangles.
- More grid rhythm, less random card piles.
- More feedback, less silence.
- More live cues, less dead surfaces.
- More premium restraint, less noisy decoration.

### Suggested visual behaviors

- Use one strong signature accent system, not many competing colors.
- Use subtle glow, gradients, soft depth, and layered surfaces sparingly.
- Use shape language that feels intentional and recognizable.
- Give key states a visual pulse or live energy.
- Use icons meaningfully, not decoratively.
- Important actions should feel touchable and confident.
- Live areas should feel responsive and active.

## Tone by audience

### Customer side

Feel:

- exciting
- welcoming
- emotionally rewarding
- easy to trust
- live and immersive

### Admin side

Feel:

- sharp
- premium
- fast
- calm under pressure
- action-first
- clearly prioritized

The admin app should not feel boring.
It should feel like a real operational command app with brand character.

## Product architecture

There are two primary app domains:

1. Customer app
2. Admin app

They share one design language, but the expression differs:

- customer = more atmospheric, inviting, destination-oriented
- admin = more structured, actionable, live, operational

## Screen families to design

You must create a full pattern system first, then apply it to screens.

### Pattern family 1: App shell

Design:

- root shell
- top app header behavior
- bottom navigation behavior
- floating and docked action areas
- safe area behavior
- scroll behavior
- loading and refresh behavior

### Pattern family 2: Dashboard

Design dashboard screens for both customer and admin:

- key hero summary
- live context
- strong next actions
- short supporting modules
- readable hierarchy

### Pattern family 3: List screens

Design list patterns for:

- bookings
- customers
- resources
- menu items
- expenses
- activity/history

Each list pattern should include:

- search
- filter chips
- segmented views where needed
- row states
- quick actions
- empty states
- loading states

### Pattern family 4: Detail and action screens

Design detail pages where the top half answers:

- state
- urgency
- next step
- primary action

Support content must come later.

This pattern is critical for:

- booking detail
- live session
- order detail
- payment detail
- resource detail

### Pattern family 5: Form and create flows

Design create flows that feel short even when multi-step.

Needed for:

- booking creation
- direct sale / POS order creation
- resource creation/edit
- menu creation/edit
- expense creation/edit

### Pattern family 6: Picker and sheet patterns

Design reusable patterns for:

- bottom sheets
- centered confirm modals
- selector sheets
- searchable item pickers
- cart review
- media selection

Rules:

- simple yes/no confirmation = centered modal
- info-rich selection or review = bottom sheet

## Product rules

### Rule 1

Action first. Details later.

### Rule 2

Live states must feel live.

### Rule 3

Long flows must feel short.

### Rule 4

Touch feedback must always feel intentional.

### Rule 5

No verbose explanation blocks unless necessary.

### Rule 6

The design must be production-grade, not demo-grade.

## Motion and feedback

Motion matters.
Polish matters.
Response matters.

Design motion rules for:

- screen transitions
- tab switching
- segmented switch
- sheet opening
- confirmation
- success feedback
- inline validation
- loading skeletons
- pull to refresh
- live update pulses

The motion style should feel:

- quick
- smooth
- premium
- light
- never gimmicky

### Feedback system

Define feedback patterns for:

- success toast
- warning toast
- error toast
- passive info toast
- inline validation
- live status chips
- pending state
- in-progress state
- completed state

## State coverage

For every important screen, design states:

- default
- loading
- empty
- error
- success
- active/live
- partial progress
- completed
- blocked

## Delivery order

Design in this order:

1. Brand direction board
2. Design tokens and foundations
3. Shell and navigation
4. Pattern library
5. Customer core screens
6. Admin core screens
7. Support CRUD screens
8. State pass
9. Motion pass
10. Polish pass

## Non-negotiables

- Keep copy short.
- Use strong iconography.
- Use stronger layout composition.
- Use more grid discipline.
- Use clearer visual hierarchy.
- Make the product feel alive.
- Make operational moments feel premium.
- Avoid generic SaaS-mobile card stacking.
- Avoid over-explaining UI.
- Avoid dead empty surfaces.

## Final outcome

The final design should make someone say:

- this feels premium
- this feels alive
- this feels smooth
- this feels like Bookinaja
- this does not feel like a generic operations app

Now design Bookinaja Mobile using this system.
